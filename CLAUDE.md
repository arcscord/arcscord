# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> A comprehensive AGENTS.md already covers commands, conventions, linting, commit style, and PR checklist. Read it first — this file focuses on architecture and patterns that require reading multiple files to understand.

## Commands

```sh
pnpm build          # full build (shared → arcscord → middleware)
pnpm typecheck      # type-check all packages
pnpm test           # run all tests (vitest)
pnpm lint           # eslint --fix (autofix by default)
pnpm lint --fix=false  # validate only, no changes
```

Run a single test file:
```sh
pnpm vitest run packages/arcscord/src/manager/command/command_manager.test.ts
```

Run tests for one package:
```sh
pnpm --filter arcscord test
```

Start the integration bot (requires `.env` with `DISCORD_TOKEN`):
```sh
pnpm test:test-bot
```

## Architecture

### Package dependency order

```
@arcscord/better-error  ←  base error class
@arcscord/error         ←  Result<T,E> helpers, depends on better-error
arcscord                ←  core framework, depends on both above
@arcscord/middleware    ←  depends on arcscord
@arcscord/cli           ←  depends on arcscord + error
```

The root `pnpm build` enforces this order. If a build fails mid-way, build packages manually in this sequence.

### Core runtime flow

```
ArcClient (extends discord.js Client)
  ├── CommandManager    → listens to interactionCreate (slash/user/message commands)
  ├── ComponentManager  → listens to interactionCreate (buttons, selects, modals)
  ├── EventManager      → wraps discord.js event listeners
  └── LocaleManager     → i18next wrapper, optional
```

`ArcClient` instantiates these managers in its constructor from `options.managers.*`. Each manager extends `BaseManager`, which gives it `this.client`, `this.logger`, and `this.trace()`.

### Interaction handling pipeline (commands and components)

```
interactionCreate
  → handleInteraction()
      → resolve handler from registry
      → create typed Context (SlashCommandContext, ButtonContext, etc.)
      → optionally deferReply (preReply option)
      → run middlewares in order
      → call handler.run(ctx)
      → call resultHandler(infos)   ← success or error Result
      → call errorHandler(infos)    ← uncaught throw or pre-run failure
```

`resultHandler` and `errorHandler` are user-overridable via `ArcClientOptions.managers.command/component`. The defaults log via `this.logger` and send an error embed using `client.getErrorMessage()`.

### Result pattern

All `run()` functions return `Result<string | true, XxxError>`:

```ts
type Result<T, E> = [null, T] | [E, null]   // from @arcscord/error

ok(true)                      // → [null, true]
ok("done")                    // → [null, "done"]
error(new CommandError({...})) // → [CommandError, null]

const [err, val] = result;
if (err) { /* handle */ }
```

Helpers: `ok`, `error`, `multiple`, `forceSafe`, `anyToError` — all from `@arcscord/error`.

### Component route matching

Components are identified by `customId`. The manager compiles each component's `customId` pattern into a route at registration time (stored in a private `WeakMap`). Dynamic segments (`:param`) are supported. On each interaction, `matchComponentRoute` finds the best match and extracts params into the context.

### Import alias

Inside `packages/arcscord/src/`, use `#/` as an alias for the source root:

```ts
import { BaseManager } from "#/base/manager/manager.class";
import { CommandError } from "#/utils/error/class/command_error";
```

Do not use relative `../../` paths for cross-directory imports within that package.

### File naming conventions

| Suffix | Contents |
|---|---|
| `.class.ts` | Class implementation |
| `.type.ts` | Type definitions only |
| `.func.ts` / `.util.ts` | Pure functions / helpers |
| `.test.ts` | Vitest tests |
| `index.ts` | Barrel re-export |

### Testing patterns

Tests use a hand-rolled mock client — there is no shared test factory from a framework. Look at existing test files for the pattern; the mock must satisfy the `ArcClient` shape that the manager under test uses.

The `arcscord/testing` module (`packages/arcscord/src/testing/`) exports mock interactions and context builders for unit tests. Import as:

```ts
import { ... } from "arcscord/testing";
```

This alias is configured in the root `vitest.config.mts` and only resolves in test context.

### Website / docs

Docs are Docusaurus under `website/`. Guide pages live in `website/docs/guide/`. API reference is generated by TypeDoc via `pnpm docs:api` then served by the Docusaurus build. Edit guide pages directly; do not edit generated API output.

### Integration test bot

`test/test_bot/` is a minimal bot used for manual Discord runtime testing. It imports `arcscord` and `@arcscord/middleware` from the monorepo source via workspace links. When changing manager or handler behavior, update the bot's handlers under `test/test_bot/src/` to reflect the new API.
